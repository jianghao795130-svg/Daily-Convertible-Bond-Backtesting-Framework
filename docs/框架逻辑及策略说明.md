# 帖子策略完整说明

## 1. 文档目的

这份文档完整说明当前项目中这套“可转债高频趋势策略”的实现逻辑。

它的来源是量化小论坛帖子：

- [可转债高频趋势策略研究——原理回测篇](https://bbs.quantclass.cn/thread/35116)

但当前项目不是直接照搬帖子里的 pandas 批处理脚本，而是把它改造成了一个**事件驱动回测框架**中的策略实现。

所以你需要分清两层：

1. 帖子原始策略思想是什么
2. 当前项目里是如何在事件驱动框架下复现它的

这份文档会把两层都讲清楚。


## 2. 策略核心思想

这套策略本质上是一个**早盘可转债趋势跟随日内策略**。

它的核心假设是：

1. 可转债早盘，尤其开盘后前 30 分钟，成交更活跃
2. 真正会继续冲的债，早盘往往会表现出：
   - 连续上涨
   - 涨速较快
   - 已形成一定趋势
   - 同时伴随一定量能
3. 买入后不做长拿，而是：
   - 一旦趋势减弱就尽快卖出
   - 一旦出现亏损苗头就止损
   - 最迟在固定时间前全部离场

所以这不是一个“低买高卖、等一整天”的策略，而是：

- 只做早盘
- 只抓已经启动的强势债
- 只拿很短时间
- 靠严格退出控制风险


## 3. 当前项目里的整体运行方式

当前项目保留的是**事件驱动框架**，不是帖子那种单日 DataFrame 循环批处理。

运行总流程如下：

1. 读取 `config.py`
2. 构建 `BacktestEngine`
3. 扫描数据文件
4. 读取并标准化行情
5. 把行情转成事件流
6. 每来一个事件就更新历史窗口
7. 每来一个事件就更新所有因子
8. 策略在当前事件上判断是否买卖
9. Broker 把信号转成订单并撮合
10. 更新现金、持仓、权益曲线
11. 输出回测结果、成交记录、净值图和评价指标

所以当前框架的本质是：

- 行情一条一条推进
- 因子一条一条更新
- 策略一条一条判断
- 订单一条一条成交


## 4. 数据频率与伪 Tick

## 4.1 你当前的数据是什么

你现在主要使用的是**分钟数据**，不是交易所真实逐笔 tick 数据。

也就是说，每根原始 K 线只有：

- 开盘价
- 最高价
- 最低价
- 收盘价
- 成交量
- 成交额

并不知道这一分钟内部真实逐笔是怎么走的。


## 4.2 为什么还要做事件驱动

帖子策略本来是高频思路，原始研究口径接近 tick 级别。

但你只有分钟数据，所以当前项目采用折中方案：

- 原始输入仍然是分钟 K 线
- 但会把每根分钟线展开成多个“伪 tick 事件”

这样做的目的不是伪造真实逐笔，而是：

- 让策略仍然按盘中事件一条条推进
- 让盘中止盈止损、趋势变化判断仍能工作


## 4.3 `synthetic_tick_seconds` 是什么意思

配置项：

```python
synthetic_tick_seconds = 60
```

它表示：

- 每隔多少秒生成一个伪 tick 事件

常见含义如下：

- `1`：每 1 秒 1 个事件，每分钟 60 个事件
- `3`：每 3 秒 1 个事件，每分钟 20 个事件
- `5`：每 5 秒 1 个事件，每分钟 12 个事件
- `15`：每 15 秒 1 个事件，每分钟 4 个事件
- `60`：每分钟只生成 1 个事件

所以你现在如果：

```python
synthetic_tick_seconds = 60
```

那么等价于：

- 读的是分钟 K 线
- 每根分钟 K 线只生成 1 个事件
- 也就是每分钟判断一次


## 4.4 伪 Tick 的价格路径怎么生成

当前代码会根据分钟线的 `OHLC` 构造一条确定性的分钟内路径。

规则是：

- 如果 `close >= open`
  - 路径假设为：`open -> low -> high -> close`
- 如果 `close < open`
  - 路径假设为：`open -> high -> low -> close`

然后在这条路径上做线性插值，生成这一分钟内部的多个事件价格。

这不是随机的。

它的特点是：

- 确定性
- 可复现
- 只是近似，不是真实逐笔


## 4.5 成交量和成交额怎么拆

分钟 K 线只有这一分钟总成交量和总成交额。

拆成伪 tick 之后，当前项目用的是：

- 成交量均分到每个伪 tick
- 成交额均分到每个伪 tick

例如一分钟总量是 600：

- 如果拆成 60 个事件，每个事件分到约 10
- 如果拆成 20 个事件，每个事件分到约 30
- 如果拆成 1 个事件，就还是 600


## 5. 帖子原始研究口径和当前项目的换算关系

这里非常重要。

帖子正文里有这行：

```python
group['tick总量'] = (group['更新时间'] - pd.to_datetime(date + ' ' + start_time)).dt.seconds / 3
```

这说明帖子研究时的参考 tick 口径其实是：

- **3 秒 1 个 tick**

不是 1 秒 1 个 tick。

所以当你现在用分钟数据时，凡是涉及“单 tick 成交量阈值”的地方，都不能简单凭感觉硬套。


## 5.1 量能阈值为什么要换算

帖子里很多规则默认是在“3 秒一个事件”的节奏下观察出来的。

但你现在如果设置：

```python
synthetic_tick_seconds = 60
```

那么你当前每个事件代表的是 60 秒，不是 3 秒。

这意味着：

- 当前每个事件上的 `volume`
- 和帖子里每个 3 秒 tick 的 `volume`

不是同一个口径。


## 5.2 当前项目怎么换算

当前策略里新增了：

```python
"tick_reference_seconds": 3,
"volume_threshold_scale_with_tick": True,
```

意思是：

- 先把你当前事件下的量能
- 折算回帖子原始的 3 秒 tick 口径
- 再去比较分档阈值

换算公式可以理解为：

```text
帖子参考口径量能 = 当前事件量能 × (当前事件秒数 / 3)
```

例如你现在：

- `synthetic_tick_seconds = 60`
- 当前某个事件的 `近平均交易量 = 100`

那么换算到帖子 3 秒口径后约等于：

```text
100 × (60 / 3) = 2000
```

所以这里严格来说不是“乘以 60”，而是：

- **乘以 60/3 = 20**


## 6. 因子体系

当前项目已经把因子拆成独立文件，统一放在：

- `cb_backtest/factors/`

配置中的名字和因子文件一一对应：

- 你在 `config.py` 里写什么因子名
- 就会去找 `cb_backtest/factors/同名文件.py`

这套设计支持三类因子：

1. 评分因子
2. 过滤因子
3. 个券择时因子


## 6.1 当前这套帖子策略主要用到的因子

核心包括：

- `收盘价`
- `涨幅`
- `近平均交易量`
- `m累计涨幅`
- `m连续增长次数`
- `m涨幅变化`
- `m涨幅变化速度`
- `m涨幅变化公式`


## 6.2 这些因子的含义

### `涨幅`

计算方式：

```text
(当前价 / 前收盘价 - 1) * 100
```

单位是：

- 百分比

例如：

- `12` 就是 `12%`
- `-0.1` 就是 `-0.1%`


### `近平均交易量`

最近 `window` 个事件的平均成交量。

注意：

- 这里的“事件”不是固定等于真实 tick
- 取决于你当前 `synthetic_tick_seconds`


### `m累计涨幅`

某个 `m` 窗口下，连续非负涨幅变化段的累计涨幅。


### `m连续增长次数`

某个 `m` 窗口下，连续非负涨幅变化持续了多少个事件。


### `m涨幅变化`

某个 `m` 窗口下的均价涨幅，相对前一个事件的变化值。


### `m涨幅变化速度`

计算方式：

```text
m累计涨幅 / m连续增长次数
```

可以理解成：

- 最近这段连续上涨过程中
- 平均每个事件的涨幅变化速度


### `m涨幅变化公式`

这是你后面补进来的一个更贴近帖子原文的因子，近似对应：

```python
((最新价 - 最新价.shift(m)) / (前收盘价 * m)) * 100
```

用途主要是：

- 做 `ma2`、`ma3` 止盈判断

## 8. 买入逻辑

当前 `config.py` 里，买入逻辑已经按帖子原文拆成配置树。

总结构是：

```text
condition1 AND condition2 AND condition3 AND condition4
```


## 8.1 condition1

帖子原文：

```python
((m3_累计涨幅 >= 0.4) & (m3_累计涨幅 < 5) & (m3_连续增长次数 > 3))
|
((m3_累计涨幅 < 0.4) & (m3_连续增长次数 > 15))
```

意思是：

- 如果 m3 累计涨幅已经比较明显
  - 那么只要连续增长次数大于 3 就可以
- 如果累计涨幅还不大
  - 那就要求连续增长次数更多，说明趋势更稳定

这个条件本质上是在判断：

- 这只债是不是已经形成了持续向上的短线趋势


## 8.2 condition2

帖子原文：

```python
m3_涨幅变化 >= m3_涨幅变化速度 * 1
and
m3_涨幅变化 < m3_涨幅变化速度 * 2.57
```

它的意思不是单纯看涨，而是看：

- 当前这一下的涨幅变化
- 相对最近一段连续上涨的平均速度
- 既不能太弱
- 也不能夸张到过热

所以它是在卡一个“合理加速区间”。


## 8.3 condition3

帖子原文：

```python
涨幅 < 12
```

含义是：

- 已经涨太多的债不追

注意当前框架里所有这些涨幅单位都是：

- 百分比

所以这里的 `12` 就是 `12%`。


## 8.4 condition4

帖子原文：

```python
近平均交易量 >= 200
```

含义是：

- 量能太小的不做
- 必须有一定成交活跃度

但这里要注意当前数据频率的口径差异：

- 如果你用真实 tick，这个阈值更接近帖子原始口径
- 如果你用分钟伪 tick，事件成交量会和真实 3 秒 tick 不同

当前项目对“仓位分档”做了自动换算，但对这个买入门槛本身，仍然建议你结合自己的数据口径进一步校准。


## 8.5 买入时间限制

帖子原文强调：

- 买入时间定在 `09:35` 之前

当前项目已补成配置：

```python
"buy_cutoff_time": "09:35:00"
```

也就是：

- 超过这个时间，该标的不再允许新开仓


## 9. 买入成交价逻辑

这是帖子策略里一个非常重要的细节。

帖子原始代码中：

1. 如果存在下一 tick
   - 买入价用 `当前价和下一价的平均值`
2. 如果不存在下一 tick
   - 买入价退化为 `当前价 * (1 + 3/1000)`

当前事件驱动框架不可能真正提前知道“未来下一根真实行情”，所以采用的是**近似复现**：

1. 如果当前事件之前已经有相邻事件
   - 用“上一事件价和当前事件价的平均值”近似
2. 如果没有相邻事件可参考
   - 用当前价上浮 `0.3%`

这不是和帖子逐 tick 完全相同，但已经尽量接近。


## 10. 仓位管理

## 10.1 基础仓位

当前配置里有：

```python
"base_position_percent": 0.10
```

表示：

- 基础目标仓位是 10%


## 10.2 量能分档放大

帖子原始逻辑里，买入时会根据当时的 `近平均交易量` 决定本次买入强度。

原始分档是：

- `> 600` -> `1.5`
- `> 1000` -> `2.0`
- `> 1500` -> `2.5`
- `> 2000` -> `3.0`

当前项目对应配置：

```python
"volume_tier_multipliers": [
    (600, 1.5),
    (1000, 2.0),
    (1500, 2.5),
    (2000, 3.0),
]
```

如果基础仓位是 10%，那么：

- 命中 `>600` 档 -> 目标仓位 15%
- 命中 `>1000` 档 -> 目标仓位 20%
- 命中 `>1500` 档 -> 目标仓位 25%
- 命中 `>2000` 档 -> 目标仓位 30%

这里默认会先把当前事件量能换算回帖子原始 3 秒 tick 口径，再比较阈值。


## 11. 卖出逻辑

当前卖出逻辑的目标是复现帖子里的五大退出方式：

1. `ma2` 止盈
2. `ma3` 止盈
3. 回落止盈
4. 涨幅过热退出
5. 止损
6. 到点强制离场

注意：

- 当前项目的卖出判断是事件驱动逐条执行的
- 和帖子里对 DataFrame 分段遍历的实现形式不同
- 但主要规则已经对上


## 11.1 `ma2` 止盈

条件是：

- 买入后涨幅 `<= 0.3%`
- 且 `m2_涨幅变化公式 < 0`

它表示：

- 虽然买入后还没明显亏
- 但两步均值变化已经转弱
- 先出掉


## 11.2 `ma3` 止盈

条件是：

- `0.3% < 买入后涨幅 <= 1%`
- 且 `m3_涨幅变化公式 < 0`
- 且 `m1_涨幅变化公式 < 0`

意思是：

- 已经有一点利润
- 但更短和更长两个尺度都出现转弱
- 提前落袋


## 11.3 回落止盈

条件是：

- 当前已经没有新的买入趋势信号
- 买入后涨幅 `> 1%`
- 当前涨幅相对持仓后最高涨幅的比例小于 `0.8`

也就是：

```text
amp_after_buy < max_rose * 0.8
```

可以理解成：

- 已经冲高过
- 现在从高点回落超过 20%
- 且没有新的继续上攻信号
- 那就先卖


## 11.4 涨幅过热退出

条件：

```python
涨幅 > 18
```

意思是：

- 当天本身已经涨太多
- 不再继续持有，直接退出


## 11.5 止损

条件是：

- `m2_涨幅变化 < 0`
- 当前没有买入信号
- 买入后涨幅 `< -0.1%`
- 持仓至少超过 1 个事件

意思是：

- 买入后已经开始亏
- 趋势也转弱了
- 而且不是刚买完立刻波动一下
- 那就止损


## 11.6 到点清仓

帖子原始思路是：

- 最迟 10 点前后全部卖掉

当前配置一般写成：

```python
"force_exit_time": "10:00:00"
```

含义是：

- 到了这个时间，如果还有持仓，就全部强平

## 15. 当前实现和帖子相比，哪些是“完全一致”，哪些是“近似”

## 15.1 基本一致的部分

这些已经基本对上：

- 买入四大条件结构
- `m3_涨幅变化` 和 `m3_涨幅变化速度` 的比较
- `ma2` / `ma3` / 回落止盈 / 止损 / 涨幅过热 / 到点离场 这些退出思路

## 15.2 近似复现的部分

这些只能尽量接近，不可能和帖子逐 tick 完全相同：

### 买入价

帖子原文是：

- 当前 tick 和下一 tick 均价

当前事件驱动里没有办法提前知道未来真实下一 tick，所以只能做近似。


### 分钟伪 tick

如果你没有真实 tick，而是用分钟数据拆伪 tick，那么：

- 分钟内路径是人工假设的
- 量能是均分的
- 不是交易所真实逐笔分布


### DataFrame 分段循环细节

帖子原始卖出是按 `signal == 1` 分段后，在 pandas 表里推进。

当前实现是：

- 按事件流逐条推进
- 用持仓状态、历史窗口和因子快照来复现

策略思想是一致的，但实现机制不同。

## 17. 一句话总结

这套策略可以概括成：

- 开盘先选出可交易的可转债候选池
- 只在早盘固定时段内寻找强势趋势债
- 通过“趋势形成 + 涨速合理 + 当日涨幅不过热 + 量能达标”来买入
- 买入后按量能决定仓位大小
- 用短线转弱、回落、止损、过热和到点清仓几套规则快速离场
- 在分钟数据场景下，通过伪 tick 机制近似复现帖子中的高频节奏

如果你现在只有分钟数据，那么它的定位应该理解为：

- **尽量贴近帖子原始高频逻辑的事件驱动近似回测版**

而不是：

- **真实逐笔 tick 的完全等价回测版**
