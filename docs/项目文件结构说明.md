# 项目文件结构说明

这份文档专门说明当前项目的目录树、父子关系，以及每一层文件大概是干什么的。

项目根目录：

```text
E:\可转债事件驱动回测框架
├─ config.py
├─ run_backtest.py
├─ cb_backtest/
├─ program/
├─ docs/
├─ data/
├─ .idea/
├─ .git/
└─ __pycache__/
```

---

## 1. 根目录

```text
E:\可转债事件驱动回测框架
├─ config.py
├─ run_backtest.py
└─ 其他子目录...
```

### `config.py`
- 整个回测框架的总配置文件。
- 这里定义：
  - 数据路径
  - 回测起止日期
  - 并行数
  - 事件频率
  - 策略参数
  - 因子配置
  - 资金、手续费、滑点等
- 它是整个项目最上层的“参数入口”。

### `run_backtest.py`
- 总入口脚本。
- 作用是按顺序串起来运行：
  1. `step1_整理数据`
  2. `step2_计算因子`
  3. `step3_选债`
  4. `step4_盘中择时交易`
  5. `step5_生成回测结果`
- 你平时如果想一键完整回测，跑它就行。

## 2. `cb_backtest/`

这是“框架核心代码目录”。

```text
cb_backtest/
├─ broker.py
├─ data.py
├─ engine.py
├─ events.py
├─ factor_config.py
├─ report.py
├─ utils.py
├─ factors/
├─ strategies/
├─ 回测合集/
└─ __pycache__/
```

### 这一层的定位
- `cb_backtest/` 是底层框架层。
- 这里放的是：
  - 账户
  - 事件定义
  - 数据读取
  - 因子基类
  - 策略基类
  - 报告生成
- `program/` 会调用这里的代码。

### `broker.py`
- 账户与成交模块。
- 管：
  - 现金
  - 持仓
  - 订单转成交
  - 资金曲线记录
- 也就是策略和账户之间的中间层。

### `data.py`
- 行情数据读取与标准化模块。
- 负责：
  - 扫描原始数据文件
  - 按 `schema` 映射字段
  - 把分钟数据转成标准事件流
  - 生成 synthetic tick

### `engine.py`
- 老的统一事件驱动引擎。
- 当前你主要跑的是 `program/step1~step5`，不是它。
- 但它仍然保留着完整的“事件驱动回测引擎”逻辑。

### `events.py`
- 定义框架里的标准事件对象。
- 包括：
  - `MarketEvent`
  - `SignalEvent`
  - `OrderEvent`
  - `FillEvent`
  - `Bar`

### `factor_config.py`
- 把配置文件里的因子写法，解析成框架可识别的结构。
- 比如：
  - 评分因子
  - 过滤因子
  - 择时因子

### `report.py`
- 回测结果输出模块。
- 负责：
  - 生成 `equity_curve.csv`
  - 生成 `fills.csv`
  - 生成 `summary.csv`
  - 画净值和回撤图

### `utils.py`
- 一些通用小工具函数。
- 比如目录创建、对象导入之类。

### `回测合集/`
- 这是你当前目录里现有的一个图片/结果集合目录。
- 更像人工整理出来的输出区，不属于主流程核心目录。

## 3. `cb_backtest/factors/`

这是“因子实现目录”。

```text
cb_backtest/factors/
├─ base.py
├─ 动量.py
├─ 分钟涨跌幅.py
├─ 均线偏离.py
├─ 收盘价.py
├─ 涨幅.py
├─ 近平均交易量.py
├─ m涨幅变化.py
├─ m涨幅变化公式.py
├─ m涨幅变化速度.py
├─ m累计涨幅.py
├─ m连续增长次数.py
├─ 成交额均值.py
├─ 成交额缩量因子.py
├─ _mean_trend_tracker.py
├─ _rolling_window_state.py
└─ __init__.py
```

### 这一层的定位
- 一个因子对应一个文件。
- `config.py` 里写什么因子名，就对应这里的同名文件。
- 这是你当前框架“因子可插拔”的核心位置。

### `base.py`
- 因子基类。
- 定义因子的统一接口。
- 现在也承载了默认“增量更新”逻辑。

### 具体因子文件
- 每个 `.py` 文件就是一个独立因子实现。
- 例如：
  - `涨幅.py`：算涨幅
  - `近平均交易量.py`：算最近平均成交量
  - `m累计涨幅.py`：算 m 窗口累计涨幅

### 下划线开头文件
- 例如：
  - `_mean_trend_tracker.py`
  - `_rolling_window_state.py`
- 这类一般不是直接给配置调用的因子，而是因子内部复用的小工具模块。

## 4. `cb_backtest/strategies/`

这是“策略实现目录”。

```text
cb_backtest/strategies/
├─ base.py
├─ factor_event_strategy.py
├─ intraday_cb_trend.py
├─ rank_rebalance.py
└─ __init__.py
```

### `base.py`
- 策略基类。
- 定义策略通用接口。

### `factor_event_strategy.py`
- 你当前主要在用的核心策略文件。
- 它负责：
  - 盘前选债
  - 候选池管理
  - 盘中因子择时
  - 买卖信号生成
  - 强平逻辑
  - 单债日内限制
- 可以理解为你现在这套“可转债事件驱动因子策略”的主策略实现。

### `intraday_cb_trend.py`
- 另一套盘中可转债趋势策略实现。
- 更像独立实验策略。

### `rank_rebalance.py`
- 排名调仓类策略。
- 和你现在主要跑的事件驱动早盘策略不是一条线。

## 5. `program/`

这是“分步流水线目录”。

```text
program/
├─ common.py
├─ step1_整理数据.py
├─ step2_计算因子.py
├─ step3_选债.py
├─ step4_盘中择时交易.py
├─ step5_生成回测结果.py
├─ step6_检查资金曲线异常.py
├─ __init__.py
└─ __pycache__/
```

### 这一层的定位
- `program/` 是当前项目的“主工作流层”。
- 它不是底层框架，而是把底层框架拼成你现在实际使用的 5+1 步流程。

### `common.py`
- 各 step 共用的公共函数。
- 负责：
  - 加载配置
  - 解析缓存路径
  - 构造 broker / strategy / data config
  - 保存 step 元数据

### `step1_整理数据.py`
- 第一步。
- 把原始分钟数据整理成标准事件缓存。
- 输出到 `data/pipeline_cache/.../step1_整理数据/`

### `step2_计算因子.py`
- 第二步。
- 对 `step1` 的标准事件缓存逐标的计算因子。
- 输出到 `step2_计算因子/`

### `step3_选债.py`
- 第三步。
- 根据盘前评分因子和过滤因子做选债。
- 输出每日候选池 `selected_symbols.pkl/csv`

### `step4_盘中择时交易.py`
- 第四步。
- 按事件驱动方式回放候选债的盘中事件。
- 执行买卖信号、生成成交和资金曲线。
- 这是最重、最慢的一步。

### `step5_生成回测结果.py`
- 第五步。
- 根据 `step4` 结果生成：
  - 成交记录
  - 资金曲线
  - 汇总表
  - 回测图

### `step6_检查资金曲线异常.py`
- 辅助检查工具，不属于主回测 1~5 流程。
- 作用是：
  - 检查资金曲线异常跳点
  - 找异常日期
  - 找最大日内回撤
  - 方便定位回测问题

## 7. `data/`

这是“运行数据与缓存目录”。

```text
data/
└─ pipeline_cache/
   └─ 可转债事件驱动因子框架示例/
      ├─ step1_整理数据/
      ├─ step2_计算因子/
      ├─ step3_选债/
      ├─ step4_盘中择时交易/
      ├─ step5_生成回测结果/
      └─ step6_检查资金曲线异常/
```

### 这一层的定位
- 这里不是源码。
- 这里是运行过程生成出来的缓存和结果。
- IDE 最好排除索引，不然很容易卡。

### `pipeline_cache/`
- 所有分步运行缓存的总目录。
- 按回测名分子目录。

### `可转债事件驱动因子框架示例/`
- 当前这份配置对应的回测缓存根目录。
- 名字来自 `config.py` 里的 `backtest_name`。

### `step1_整理数据/`
- 存 step1 生成的标准事件缓存。
- 一般是每个标的一个文件。

### `step2_计算因子/`
- 存 step2 生成的因子缓存。
- 后面 `step3` 和 `step4` 都会读它。

### `step3_选债/`
- 存盘前选债结果。
- 核心文件是：
  - `selected_symbols.pkl`
  - `selected_symbols.csv`

### `step4_盘中择时交易/`
- 存盘中事件回放后的原始结果。
- 核心文件是：
  - `equity_curve.pkl`
  - `fills.pkl`

### `step5_生成回测结果/`
- 存最终给你看的报表结果。
- 核心文件是：
  - `equity_curve.csv`
  - `fills.csv`
  - `summary.csv`
  - `equity_drawdown.png`

### `step6_检查资金曲线异常/`
- 存资金曲线异常排查结果。
- 比如：
  - 单点异常
  - 日内最大回撤
  - 异常摘要

## 10. `__pycache__/`

```text

└─ Python 编译缓存
```

- 这是 Python 自动生成的字节码缓存目录。
- 跟逻辑本身无关。
- 删了也会重新生成。

## 11. 你可以怎么理解“父子关系”

最重要的父子关系其实是这条：

```text
config.py
  └─ run_backtest.py / program/*.py
       └─ cb_backtest/ 底层框架
            └─ data/ 运行结果与缓存
```

也可以按“工作流”理解成：

```text
原始数据
  └─ step1_整理数据
       └─ step2_计算因子
            └─ step3_选债
                 └─ step4_盘中择时交易
                      └─ step5_生成回测结果
                           └─ step6_检查异常（可选辅助）
```

---

## 12. 一句话总结

这个项目可以分成三层：

### 第一层：底层框架层
- `cb_backtest/`
- 负责账户、事件、数据、策略、因子、报告这些基础能力

### 第二层：分步流程层
- `program/`
- 负责把底层框架拼成你现在实际使用的 5+1 步回测工作流

### 第三层：运行缓存与结果层
- `data/`
- 负责保存每一步产生的缓存和最终回测结果
